## 失败拆解

### 1. 工具 / 协议失败

Agent 没有总是调用所有 required tools，并且有一次工具调用 schema 不合法。

代表 case：

- `workspace-project-map-001`
  - 失败字段：`tool_success`、`output_sections_success`
  - Run：`run_20260630_211500_dc77ab7c`
  - 证据：`git_inspector` 调用缺少必需的 `operation`，触发 `tool_validation_failed`。

- `toolgateway-workspace-verification-001`
  - 失败字段：`tool_success`、`trace_event_success`
  - Run：`run_20260630_212835_98c17249`
  - 证据：trace 里只有 `run_started`、`model_called`、`skill_selected`、`run_finished`，没有 `tool_called`。
  - 解释：Agent 根据推断或上下文直接回答，没有先调用 `workspace_files` 和 `verification_runner`。

- `runtime-tool-workspace-code-search-001`
  - 失败字段：`tool_success`、`keyword_success`
  - Run：`run_20260630_212517_f918794c`
  - 证据：`code_search` 报 `[WinError 2]`，随后 Agent 降级使用 `file_reader`。
  - 解释：工具降级部分生效，但因为没有成功的 `code_search` `tool_called` 事件，required-tool 指标失败。

### 2. 工具预算失败

代表 case：

- `workspace-architecture-chain-001`
  - 失败字段：`tool_success`、`keyword_success`、`output_sections_success`
  - Run：`run_20260630_211552_37533964`
  - 证据：trace 以 `Maximum tool calls exceeded: 12` 结束。
  - 解释：Agent 读文件过多，没有先做紧凑的架构探测。这是规划和工具预算问题。

### 3. 输出结构弱匹配

很多 case 的 `output_sections_success=False` 是因为输出标题和 `expected_output_sections` 的精确字符串不同。

例子：

- case 期望 `已读取文件`，回答用了 `工作区事实探测结果`。
- case 期望 `Unconfirmed Items`，回答有时使用了纯中文的不确定事项章节。

解释：这类失败应该人工复核。它们可以推动输出结构一致性，但不足以单独证明语义质量失败。

### 4. 本地测试环境失败

用户完整 PowerShell 日志显示，真实 eval suites 之前的 pytest 失败来自：

```text
PermissionError: C:\Users\32834\AppData\Local\Temp\pytest-of-admin
```

这是本地 pytest 临时目录权限问题，不代表 real eval runner 失败。后续本地测试检查建议使用：

```powershell
python -m pytest --basetemp .\.pytest-tmp -q
```

如果被评估的 workspace 依赖自己的 `.venv`，workspace-specific verification 命令也应该用那个解释器。

## 建议的下一步工程工作

1. 加强 prompt 或 Skill 中的工具 schema 指导，尤其是 `git_inspector`。
2. verification 请求在回答前必须至少尝试 `workspace_files` 或 `verification_runner`。
3. 改进架构审查规划，让 Agent 先用定向搜索收敛范围，再读大文件。
4. 决定输出章节标题到底是严格产品契约，还是只作为人工 rubric 的参考。
5. 改进 `code_search` 在 Windows 上的可靠性，并记录它依赖的路径/命令要求。
6. 所有 runbook 都补充 workspace-local pytest 临时目录建议。

## 面试总结

这次运行的价值在于它不是“假装全绿”。它证明真实 eval 链路可以跑通，同时暴露出 Agent 行为仍需要工程化的地方：
