# 开题报告 Skill

## 什么时候使用

当用户需要准备研究生开题报告、论文选题、研究计划、研究背景、技术路线、可行性分析、风险分析、导师汇报材料或下一步资料清单时，使用这个 Skill。

这个 Skill 可以服务不同专业方向。通信工程、工业遮挡失联、Spring Boot 等内容只能作为用户背景、资料内容或后续专门 Skill 的知识来源，不能写死进核心 Runtime。

## 输入信息检查

开始前先判断用户是否已经提供以下信息：

- 研究方向或专业背景
- 学校开题报告模板或格式要求
- 导师反馈或已有修改意见
- 已有论文题目、草稿或想法
- 参考文献、调研资料或 Word 文档
- 论文类型，例如理论研究、系统实现、工程应用或综述
- 技术栈、实验条件或数据来源
- 截止时间和本次希望完成的范围

如果关键信息缺失，不要假装已经知道。可以先基于已有信息给出初稿框架，同时明确列出待确认问题。

## 工作流程

1. 判断用户当前需求属于选题、提纲、研究背景、研究问题、技术路线、可行性分析、风险分析、完整开题草稿还是修改润色。
2. 阅读用户提供的背景、草稿、模板或导师意见，提取约束和关键目标。
3. 如果用户提供了 Word 文档路径，可以请求使用 `docx_reader` 读取正文。
4. 如果需要事实依据、文献依据或已有资料支撑，可以请求使用 `rag_search` 检索相关片段。
5. 根据用户需求组织输出结构，不要默认一次性生成完整开题报告。
6. 输出时说明当前版本基于哪些已知信息，以及哪些地方仍是假设。
7. 如果用户明确要求保存为文件，可以请求使用 `markdown_writer` 写出 Markdown 草稿。
8. 最后给出下一步建议，例如需要补充哪些资料、下一轮应该先改哪一部分。

## 输出类型

根据用户需求选择合适输出，不要每次都输出完整大稿。

常见输出包括：

- 候选论文题目
- 开题报告目录结构
- 研究背景与意义
- 国内外研究现状梳理
- 研究目标与研究问题
- 技术路线或系统方案
- 可行性分析
- 创新点与难点
- 风险分析与备选方案
- 下一步资料清单
- Markdown 草稿文件

## 选题输出要求

如果用户要选题，至少给出：

- 2 到 3 个候选题目
- 每个题目的研究对象
- 每个题目的实现难度
- 每个题目的资料可获得性
- 推荐优先级和取舍理由

不要只给一个看起来很正式的题目，也不要编造导师已经认可。

## 研究背景输出要求

如果用户要研究背景，优先说明：

- 问题场景是什么
- 为什么这个问题值得研究
- 当前方法或系统有什么不足
- 用户已有资料能支持哪些判断
- 哪些判断还需要文献或数据支撑

没有资料时，要明确说明“这是基于已有背景的初稿，不是文献结论”。

## 研究问题输出要求

如果用户要研究问题，尽量写成可研究、可验证的问题，例如：

- 要解决什么具体问题
- 输入和输出是什么
- 约束条件是什么
- 如何判断方案有效
- 哪些问题属于本文范围之外

## 技术路线输出要求

如果用户要技术路线，优先组织为：

1. 数据或资料来源
2. 核心方法或系统模块
3. 处理流程
4. 实验或验证方式
5. 预期输出

如果涉及工程实现，可以说明前端、后端、数据库、算法模块或接口关系，但不要把工程实现写成核心 Runtime 代码。

## 可行性分析输出要求

可行性分析至少考虑：

- 技术可行性
- 资料或数据可行性
- 时间可行性
- 工程实现可行性
- 风险与替代方案

不要只写“本课题具有可行性”这种空话。

## 风险分析输出要求

风险分析至少包含：

- 资料不足风险
- 实验或数据获取风险
- 工程实现复杂度风险
- 论文范围过大风险
- 备选降级方案

## 工具使用原则

- 只有用户提供 Word 文档路径，或明确要求读取 Word 文档时，才请求 `docx_reader`。
- 只有需要基于资料、文献、笔记或已有知识库找依据时，才请求 `rag_search`。
- 只有用户明确要求生成、保存或导出文件时，才请求 `markdown_writer`。
- 不要为了显得复杂而调用工具；能直接回答的小问题可以直接回答。
- 工具只负责动作，不能替代 Agent 判断任务目标和输出结构。

## 约束

- 不编造参考文献、实验结果、导师意见或学校要求。
- 不把用户没有确认的研究方向当成事实。
- 不主动写文件，除非用户明确要求保存或导出。
- 默认使用 Markdown 输出，除非用户要求其他格式。
- 不把具体专业、具体论文题目或具体工程场景写死进 Runtime。
- 不执行 `scripts/` 下的脚本；脚本执行必须等后续 ToolGateway、权限检查、用户确认和 Trace 支持。

## 完成检查项

完成任务前检查：

- 已明确用户本次真正想完成的内容。
- 已说明当前回答基于哪些已知信息。
- 如果信息不足，已列出待确认问题。
- 如果涉及选题，已给出题目方向或取舍理由。
- 如果涉及开题结构，已覆盖背景、问题、目标、路线、可行性和风险。
- 如果使用资料依据，已说明资料来源或检索依据。
- 如果用户要求保存文件，已准备调用 `markdown_writer`。
- 已给出下一步行动建议。
