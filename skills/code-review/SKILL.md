# 代码审查 Skill

这个 Skill 面向中文用户，用于审查一个本地项目或仓库。核心原则：先读项目、再追链路、最后下结论；所有重要判断都要有文件证据，不要提前总结。

## 工作流程

1. 先用 `workspace_files` 做项目地图：目录分层、配置文件、入口文件、核心源码、测试目录、脚本命令和文档位置。
2. 用 `file_reader` 阅读最小必要集合：README/项目说明、依赖与配置、入口文件、核心模块、关键工具或服务、现有测试。不要只读文件名就判断。
3. 形成阅读计划并持续补读：每次发现入口、调用方、被调用方、测试或配置缺口，都继续读取关键文件；如果关键文件没读到，要放进“未确认事项”。
4. 至少追一条关键链路：从用户入口或 CLI/API 入口开始，追到核心业务逻辑、工具/服务调用、状态或数据流、错误处理，再对照相关测试。
5. 输出事实时标注证据：重要结论要绑定具体文件路径，能定位行号时给出行号；区分“已确认事实”“基于代码的推断”“未确认事项”。
6. 做风险分级：按 P0/P1/P2 或 高/中/低说明影响、证据、触发条件和修复方向。没有证据的猜测不要放进确定问题。
7. 检查测试缺口：说明已有测试覆盖哪些行为，缺少哪些关键路径、边界条件、错误路径、回归风险，以及建议运行哪些验证命令。
8. 用户要求保存时，用 `markdown_writer` 写出中文审查报告。

## 输出要求

- 默认用中文回答。
- 先给高信号结论，再给证据和建议；不要用泛泛的“整体不错”“建议加强测试”代替具体分析。
- 不要编造文件、测试结果或项目事实；没读到就说明没读到。
- 如果信息不足，优先请求工具读取，而不是凭印象下判断；关键文件未读时，不要提前总结。
- 建议要可执行，例如指出应补的测试、应拆的模块或应验证的命令。
- 审查不是摘要。必须解释代码如何运行、风险为什么成立、测试为什么不足。

## 深度检查清单

| 检查项 | 必须回答的问题 |
| --- | --- |
| 已读取文件清单 | 读了哪些文件？每个文件用于确认什么？ |
| 项目地图 | 入口、配置、核心模块、测试、脚本分别在哪里？ |
| 关键链路 | 一个主要请求、命令或任务如何从入口流到核心逻辑并返回结果？ |
| 证据 | 每个主要结论对应哪个文件或代码现象？ |
| 风险分级 | 问题影响多大、是否可复现、优先级是什么？ |
| 测试缺口 | 现有测试覆盖什么，还缺哪些行为、边界和失败路径？ |
| 未确认事项 | 哪些判断因为文件、运行结果或上下文不足还不能确认？ |
