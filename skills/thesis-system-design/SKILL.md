# 论文系统设计 Skill

## 什么时候使用

当用户需要设计工程实现型论文的系统方案、系统架构、功能模块、业务流程、接口、数据库表、实验验证方案、Spring Boot 工程计划或论文系统设计章节时，使用这个 Skill。

这个 Skill 可以服务不同专业方向。通信工程、工业遮挡失联、定位、通信状态感知、Spring Boot 等内容可以作为用户背景和真实落地案例，但不能写死进核心 Runtime。

## 输入信息检查

开始前先判断用户是否已经提供以下信息：

- 论文题目或研究方向
- 系统要解决的核心问题
- 使用场景和目标用户
- 主要功能需求
- 技术栈，例如 Spring Boot、Vue、MySQL、Redis、Python 算法服务等
- 数据来源，例如人工录入、传感器数据、仿真数据、文献数据或实验数据
- 系统输入和输出
- 论文学校模板或章节要求
- 是否需要接口、数据库表、模块图、流程图或实现计划
- 截止时间和本次希望完成的范围

如果关键信息缺失，不要假装已经知道。可以先给出合理的系统设计框架，并明确列出待确认问题。

## 工作流程

1. 判断用户当前需要的是系统总体方案、模块拆分、数据库设计、接口设计、流程设计、验证方案、章节草稿还是工程实现计划。
2. 提取用户提供的研究方向、论文目标、工程约束和技术栈。
3. 如果用户提供 Word 模板、草稿或需求文档，可以请求使用 `docx_reader` 读取正文。
4. 如果需要参考已有资料、文献依据或项目笔记，可以请求使用 `rag_search` 检索相关片段。
5. 先明确系统边界：系统要解决什么问题，不解决什么问题。
6. 再拆分模块：每个模块说明职责、输入、输出和依赖。
7. 再设计数据流：说明数据从哪里来、如何处理、如何展示或保存。
8. 再设计接口和数据库：只给和当前需求相关的最小必要设计，不一次性堆太多表和接口。
9. 最后把工程设计映射成论文可以写的章节结构。
10. 如果用户明确要求保存，可以请求使用 `markdown_writer` 写出 Markdown 草稿。

## 系统需求分析

系统需求分析至少说明：

- 目标用户是谁
- 系统解决什么具体问题
- 输入数据是什么
- 输出结果是什么
- 必须支持哪些核心功能
- 哪些功能属于后续扩展，不放进第一版
- 评价系统是否有效的标准是什么

不要只写“系统应具有良好的可扩展性”这种空泛句子。

## 模块设计方法

拆模块时优先使用“职责清楚、边界明确”的方式。每个模块至少说明：

- 模块名称
- 模块职责
- 输入
- 输出
- 依赖的其他模块
- 对论文的价值

常见模块可以包括：

- 用户与权限模块
- 数据采集或数据导入模块
- 数据管理模块
- 算法分析或状态感知模块
- 结果展示模块
- 报告导出模块
- 系统管理模块

模块不是越多越好。第一版应该优先保证论文能讲清楚、工程能实现。

## 数据库设计方法

设计数据库表时，先从实体关系出发，而不是直接堆字段。优先确认：

- 系统有哪些核心实体
- 实体之间是什么关系
- 哪些数据需要长期保存
- 哪些只是临时计算结果
- 哪些字段服务论文展示或实验分析

输出数据库表时，至少包含：

- 表名
- 表用途
- 核心字段
- 字段含义
- 主键或关键索引
- 和其他表的关系

不要一次性生成过多无用字段。

## 接口设计方法

设计接口时，优先围绕用户流程和模块交互。每个接口至少说明：

- 接口名称
- 请求方法
- 路径
- 输入参数
- 返回结果
- 所属模块
- 使用场景

如果用户没有明确要求，不要一次性输出完整 Swagger 风格的大量接口。

## 论文写作映射

工程设计要能转化成论文内容。输出时尽量说明：

- 系统总体架构可以写进“系统总体设计”
- 模块职责可以写进“功能模块设计”
- 数据库表可以写进“数据库设计”
- 接口和流程可以写进“系统详细设计”
- 实验或验证方式可以写进“系统测试与结果分析”
- 工程实现难点可以写进“关键技术实现”

不要只给程序员视角的开发清单，也要给论文写作视角的章节安排。

## 工具使用原则

- 只有用户提供 Word 文档路径，或明确要求读取 Word 文档时，才请求 `docx_reader`。
- 只有需要参考已有资料、文献、笔记或知识库时，才请求 `rag_search`。
- 只有用户明确要求保存、导出或生成文件时，才请求 `markdown_writer`。
- 能直接解释的小范围系统设计问题，不需要强行调用工具。
- 工具只负责动作，不能替代 Agent 判断系统边界和设计取舍。

## 约束

- 不直接创建 Spring Boot 项目。
- 不编造用户没有确认的技术栈、数据库、算法或实验数据。
- 不把通信工程作为唯一方向写死。
- 不主动写文件，除非用户明确要求保存或导出。
- 不把系统设计写成泛泛而谈的模板话。
- 不执行 `scripts/` 下的脚本；脚本执行必须等后续 ToolGateway、权限检查、用户确认和 Trace 支持。

## 完成检查项

完成任务前检查：

- 已明确用户本次要做的是总体方案、模块、数据库、接口、流程、验证还是章节草稿。
- 已说明系统要解决的问题和系统边界。
- 已给出模块职责，而不是只列模块名称。
- 如果涉及数据库，已说明表用途、核心字段和关系。
- 如果涉及接口，已说明路径、参数、返回结果和使用场景。
- 已把工程设计映射到论文章节。
- 如果信息不足，已列出待确认问题。
- 如果用户要求保存文件，已准备调用 `markdown_writer`。
- 已给出下一步实现或写作建议。
